# Contributing to CS9618 Open Learning

Thank you for helping make computer science education more accessible. Contributions support **UN SDG 4 (Quality Education)** when they are accurate, inclusive, and easy to reuse.

## Principles

1. **Syllabus first** — Map changes to official **9618** section numbers (e.g. 9.2 Algorithms). If Cambridge updates wording, prefer the **latest syllabus** for your exam series.
2. **Clarity for ESL** — Short sentences, define jargon on first use, avoid idioms where possible.
3. **No past-paper copyright** — Do not paste full past-paper questions or mark schemes from Cambridge. Write **original** exam-style questions “in the style of” the syllabus command words.
4. **MIT license** — By contributing, you agree your contributions are licensed under the [LICENSE](LICENSE) in this repository.

## Ways to contribute

- Fix typos, links, or syllabus references in any `*.md` file.
- Add **worked examples** or **diagrams** (ASCII, Mermaid, or linked images you have rights to use).
- Add **original** short-answer or structured questions in `exercises.md`.
- Improve **real-world applications** or **further reading** (open-access sources preferred).
- Translate or adapt a unit for a local context (open a PR or issue to coordinate).

## Content checklist (subtopic edits)

- [ ] Headings use `#`, `##`, `###` consistently.
- [ ] **Learning objectives** in the parent unit `README.md` still match the subtopic.
- [ ] Include where helpful: **Exam Tip**, **Common Mistakes**, **Further Reading**.
- [ ] Pseudocode follows ideas in the official **Pseudocode Guide** (this repo includes a PDF copy under `resources/references/originals/`).
- [ ] No scanned or full-text third-party textbook extracts.

## Git workflow

1. Fork the repository (when hosted on GitHub) or work on a branch.
2. One logical change per pull request (e.g. “Unit 8: normalisation examples”).
3. In the PR description, list **syllabus sections** touched and any **assumptions** (e.g. exam series year).

## Local preview

Any Markdown viewer works. For GitHub-flavoured rendering, push to a branch and open a PR.

## Code of conduct

Be respectful and constructive. Assume good intent; prioritise learner understanding over debate.

## Questions

Open an issue with label suggestions from [.github/github-labels.md](.github/github-labels.md), such as `curriculum` or `good first issue`.
